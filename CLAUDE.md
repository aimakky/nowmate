# YVOICE — Claude Code プロジェクトルール

このファイルは YVOICE（ワイボイス、略：ワイボ）リポジトリの永続ルール集です。
Claude Code はこのリポジトリで作業するとき、ここに書かれているルールを必ず守ってください。

## サービス概要

**YVOICE = Your Voice Online**

20歳以上限定の大人向けゲーム通話コミュニティ。自分の声でオンラインのゲーム仲間とつながり、安心して遊べる場所。
電話番号認証・本人確認 (Stripe Identity)・Trust Tier 制度で民度を守ります。出会い系ではなくゲーム仲間と遊ぶための通話コミュニティ。

主要機能: いますぐ村 / ギルド / 通話ルーム / フレンド / タイムライン / 安心ガイド。


> 旧名称 (samee / nowmate / nowmatejapan / 自由村) はリポジトリ root ディレクトリ名・
> ライブドメイン・GitHub repo 名・Vercel project 名・Supabase project 名にだけ
> 残置されています。これらはサービス停止リスクのため別フェーズで移行します。
> ユーザー可視テキストは原則として YVOICE に統一してください。

## 作業報告ルール（永続）

すべての **調査・設計・実装・修正・バグ対応** の最後に、以下の **3 つ** を必ず出力すること。

1. **ChatGPT 相談用プロンプト**
2. **次回 Claude Code へコピペする用プロンプト**
3. **小学生でも分かる説明プロンプト**

目的：

- ユーザーが画面のスクリーンショットを撮らなくても、ChatGPT へそのまま貼って相談できるようにすること
- ChatGPT で判断したあと、その結果を次の Claude Code 作業指示としてそのまま使えるようにすること
- 専門知識がなくても、ユーザー本人がその場で「今回何が進んだか」「次に何をするか」を理解できるようにすること

3 つすべてを出さない作業報告は不完全とみなす。

#3 は **ChatGPT に投げるためのプロンプトではない**。Claude Code 自身が、その場で読めば分かる説明を直接書く。コードフェンスで囲まない。

### 出力スタイル（共通ルール）

- **表 (table) は使わない**。スマホからのコピーが崩れるため。
- 箇条書き中心、1 行は短く、改行を多めに。
- 各プロンプトは ` ```text ` のコードフェンスで囲む（コピペしやすくするため）。
- 空欄のテンプレートをそのまま渡さない。各項目は必ずその時の状況で埋める。

### 1. ChatGPT 相談用プロンプト（必須フォーマット）

ユーザーが画面を写真で撮らなくても、そのまま ChatGPT へ貼って相談できる「相談プロンプト」として出力する。
単なるまとめではなく、ChatGPT に質問して判断を仰ぐ完成形のプロンプトにする。

見出しは必ず **【ChatGPT相談用プロンプト】** とすること。

必ず以下の形式・コードフェンス（` ```text `）付きで出力すること：

````
```text
以下は、Claude Codeで進めたYVOICE開発の作業状況です。
内容を確認して、次に何をすべきか分かりやすく判断してください。

【現在の状況】
- 

【完了したこと】
- 

【今回、何を改善したのか】
- 

【今回、何を実装したのか】
- 

【未実装の内容】
- 

【今回変更したファイル】
- 

【エラーの有無】
- build：
- lint：
- TypeScript：
- runtime：
- DB：
- 環境変数：
- 未実行のものがあれば「未実行」と明記

【次に選ぶべき選択肢】
- A案：
- B案：
- C案：

【Claude Codeのおすすめ】
- 

【確認したいこと】
- 

上記を踏まえて、
1. 今の状態は安全に進んでいるか
2. 今回の実装内容は問題ないか
3. 追加実装する前にコミット＆pushすべきか
4. 次はA/B/Cのどれを選ぶべきか
5. 危険な変更や後回しにすべき作業はあるか
6. 次にClaude Codeへ貼るべき指示プロンプト

を分かりやすく教えてください。
```
````

ルール：

- 各 `【...】` は空欄で渡さない。その時の状況で必ず埋める。
- エラーが「未実行」の項目はそのまま `未実行` と書く。
- 該当なしの項目は `なし` と書いて空欄にしない（ChatGPT が判断しやすくなる）。
- 「【次に選ぶべき選択肢】」は最低 2 つ、できれば 3 つ提示する。
- 「【今回、何を改善したのか】」と「【今回、何を実装したのか】」は別物として書き分ける
  （改善＝既存を直した・整えた、実装＝新しく追加した）。両方ない場合は片方を `なし` と書く。

### 2. 次回 Claude Code へコピペする用プロンプト（必須フォーマット）

ChatGPT 相談後、ユーザーがそのまま Claude Code へ貼れるように、次の作業指示プロンプトを作成する。

必ず以下の形式・コードフェンス（` ```text `）付きで出力すること：

````
```text
前回の作業状況を踏まえて、次の作業を安全に進めてください。

【前回までの状況】
- 

【今回やってほしいこと】
- 

【今回、何を改善したのか】
- 

【今回、何を実装したのか】
- 

【未実装の内容】
- 

【次に何を実装しようとしているのか】
- 

【絶対に守ること】
- 既存機能を壊さない
- 既存UIを大きく崩さない
- DB変更が必要な場合は勝手に実行せず、SQL案を先に提示する
- RLS、Webhook、認証、環境変数まわりの変更は必ず影響範囲を確認する
- Supabase、Stripe、Twilio、LiveKitなど外部サービス連携を壊さない
- サービス名 YVOICE / Your Voice Online 関連の文言を壊さない
- 下部タスクバー、TL、フレンド、ゲーム村、いますぐ村、ギルド、チャット、通知、プロフィール、本人確認済みバッジを壊さない
- build / lint / TypeScript確認を行う
- 実装後は最後に以下の3つを必ず出す
  1. ChatGPT相談用プロンプト
  2. 次回Claude Codeへコピペする用プロンプト
  3. 小学生でも分かる説明プロンプト

【確認してほしいこと】
- 

【実装後の報告】
- 変更したファイル一覧
- 実装した内容
- 改善した内容
- 未実装の内容
- 次に実装予定の内容
- エラーの有無
- build / lint / TypeScript確認結果
- 次に進むべき作業
```
````

`【前回までの状況】` `【今回やってほしいこと】` `【今回、何を改善したのか】` `【今回、何を実装したのか】` `【未実装の内容】` `【次に何を実装しようとしているのか】` `【確認してほしいこと】` の 7 つは、その時の状況に応じて Claude Code 側で具体的に埋めること。空欄のまま渡さない。

### 3. 小学生でも分かる説明プロンプト（必須フォーマット）

これは **ChatGPT に投げるためのプロンプトではない**。
Claude Code 自身が、今回やったことと次にやろうとしていることを、小学生でも分かる言葉でその場で書く。
ユーザーがその場で読んで、技術的な前提なしに「今これがどうなった」「次にこれをする」を理解できるようにする。

見出しは必ず **【小学生でも分かる説明プロンプト】** とすること。

形式：

- コードフェンスで囲まない（コピペ用ではなく読み物のため）
- 構成は以下 2 つの小見出しを必ず入れる
  - **今回やったこと**
  - **次にやろうとしていること**
- 各小見出しの下は、3 〜 5 行のかんたんな日本語で説明する箇条書き
- 最後に「これで何がうれしくなったの？」を 1 〜 2 行で書く

書き方ルール：

- むずかしい言葉（API・コミット・build・TypeScript・RLS・Webhook・state・remount など）は、子どもでも分かる言いかえに直す
- たとえ話を 1 つは入れる（例：「お部屋の電気を切り替えるみたいに」「ノートのページをめくるみたいに」）
- ファイル名・関数名・コード片は書かない
- 1 トピックにつき 1 〜 3 個の箇条書きで十分（多すぎると小学生説明にならない）
- えらそうな言い方や、専門家っぽい説明はしない
- 句読点と改行を多めにして、読みやすくする

例（雰囲気のサンプル）：

```
【小学生でも分かる説明プロンプト】

今回やったこと
- アプリの「フォローしている人」「フォロワー（自分をフォローしている人）」を見るためのボタンが、押しても動かなかったんだ。
- ボタンに「押されたよ！」のしくみが入ってなかったから、押されても何も起きなかった。
- そこに「押されたら一覧を見せる」しくみを入れて、ちゃんと切り替わるようにしたよ。

次にやろうとしていること
- スマホで実際にアプリを開いて、ボタンが本当にちゃんと動くか目で見てたしかめる予定。

これで何がうれしくなったの？
- 自分がだれをフォローしているか・だれがフォローしてくれているかが、ちゃんと見られるようになる。
```

### 適用対象

- 通常の機能実装
- バグ修正
- 設計フェーズの調査
- env / RLS / Webhook 等の運用作業
- 1 ターンで完結する小さな修正でも適用する（簡略化しない）

### 適用しなくていい例外

- ユーザーが「報告フォーマット不要」「短く返して」と明示した場合のみスキップ可
- 単なる質問への回答（コード変更を伴わない）は任意
