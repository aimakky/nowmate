# YVOICE — Claude Code プロジェクトルール

このファイルは YVOICE（ワイボイス、略：ワイボ）リポジトリの永続ルール集です。
Claude Code はこのリポジトリで作業するとき、ここに書かれているルールを必ず守ってください。

## サービス概要

**YVOICE = Your Voice Online**

20歳以上限定の大人向けゲーム通話コミュニティ。自分の声でオンラインのゲーム仲間とつながり、安心して遊べる場所。
電話番号認証・本人確認 (Stripe Identity)・Trust Tier 制度で民度を守ります。出会い系ではなくゲーム仲間と遊ぶための通話コミュニティ。

主要機能: いますぐ村 / ギルド / 通話ルーム / フレンド / タイムライン / 安心ガイド。


> 旧名称 (samee / nowmate / nowmatejapan / 自由村) はリポジトリ root ディレクトリ名・
> ライブドメイン・GitHub repo 名・Vercel project 名・Supabase project 名にだけ
> 残置されています。これらはサービス停止リスクのため別フェーズで移行します。
> ユーザー可視テキストは原則として YVOICE に統一してください。

## 作業報告ルール（永続）

すべての **調査・設計・実装・修正・バグ対応** の最後に、以下の **3 つ** を必ず出力すること。

1. **ChatGPT 相談用プロンプト**
2. **次回 Claude Code へコピペする用プロンプト**
3. **簡潔で分かりやすい説明プロンプト**

目的：

- マッキーさん（開発依頼者本人）が画面のスクリーンショットを撮らなくても、ChatGPT へそのまま貼って相談できるようにすること
- ChatGPT で判断したあと、その結果を次の Claude Code 作業指示としてそのまま使えるようにすること
- マッキーさんが短時間で「今回何が進んだか」「次に何をするか」「自分にとって何が分かりやすくなったか」「YVOICE 利用者にとって何が便利になったか」を理解できるようにすること

3 つすべてを出さない作業報告は不完全とみなす。

#3 は **ChatGPT に投げるためのプロンプトではない**。Claude Code 自身が、マッキーさんに向けて簡潔に分かりやすく直接書く。コードフェンスで囲まない。

### 出力スタイル（共通ルール）

- **表 (table) は使わない**。スマホからのコピーが崩れるため。
- 箇条書き中心、1 行は短く、改行を多めに。
- 各プロンプトは ` ```text ` のコードフェンスで囲む（コピペしやすくするため）。
- 空欄のテンプレートをそのまま渡さない。各項目は必ずその時の状況で埋める。

### 1. ChatGPT 相談用プロンプト（必須フォーマット）

ユーザーが画面を写真で撮らなくても、そのまま ChatGPT へ貼って相談できる「相談プロンプト」として出力する。
単なるまとめではなく、ChatGPT に質問して判断を仰ぐ完成形のプロンプトにする。

見出しは必ず **【ChatGPT相談用プロンプト】** とすること。

必ず以下の形式・コードフェンス（` ```text `）付きで出力すること：

````
```text
以下は、Claude Codeで進めたYVOICE開発の作業状況です。
内容を確認して、次に何をすべきか分かりやすく判断してください。

【現在の状況】
- 

【完了したこと】
- 

【今回、何を改善したのか】
- 

【今回、何を実装したのか】
- 

【未実装の内容】
- 

【今回変更したファイル】
- 

【エラーの有無】
- build：
- lint：
- TypeScript：
- runtime：
- DB：
- 環境変数：
- 未実行のものがあれば「未実行」と明記

【次に選ぶべき選択肢】
- A案：
- B案：
- C案：

【Claude Codeのおすすめ】
- 

【確認したいこと】
- 

上記を踏まえて、
1. 今の状態は安全に進んでいるか
2. 今回の実装内容は問題ないか
3. 追加実装する前にコミット＆pushすべきか
4. 次はA/B/Cのどれを選ぶべきか
5. 危険な変更や後回しにすべき作業はあるか
6. 次にClaude Codeへ貼るべき指示プロンプト

を分かりやすく教えてください。
```
````

ルール：

- 各 `【...】` は空欄で渡さない。その時の状況で必ず埋める。
- エラーが「未実行」の項目はそのまま `未実行` と書く。
- 該当なしの項目は `なし` と書いて空欄にしない（ChatGPT が判断しやすくなる）。
- 「【次に選ぶべき選択肢】」は最低 2 つ、できれば 3 つ提示する。
- 「【今回、何を改善したのか】」と「【今回、何を実装したのか】」は別物として書き分ける
  （改善＝既存を直した・整えた、実装＝新しく追加した）。両方ない場合は片方を `なし` と書く。

### 3. 簡潔で分かりやすい説明プロンプト（必須フォーマット）

これは **ChatGPT に投げるためのプロンプトではない**。
Claude Code 自身が、この形式に沿って、今回の作業内容と次回予定を **マッキーさんに向けて** 簡潔に分かりやすく直接書く。

見出しは必ず **【簡潔で分かりやすい説明プロンプト】** とすること。

必ず以下の形式・コードフェンス（` ```text `）付きで出力すること：

````
```text
今回の作業内容を、マッキーさんが短時間で理解できるように説明します。

【今回やったこと】
- 

【改善されたこと】
- 

【マッキーさんにとって分かりやすくなったこと】
- 

【YVOICE利用者にとって便利になったこと】
- 

【まだ終わっていないこと】
- 

【次にやる予定のこと】
- 

【注意点】
- 

【エラーや問題】
- 

【今やるべきこと】
- 
```
````

書き方ルール：

- 各 `【...】` は空欄で渡さない。該当なしの項目は `なし` と必ず書く。
- 1 項目につき 1 〜 3 行の短い箇条書き。長文・段落説明は禁止。
- 専門用語（API・コミット・build・TypeScript・RLS・Webhook・state・remount 等）は使わず、平易な日本語に置き換える。
- ファイル名・関数名・コード片は書かない（マッキーさんは中身ではなく結果を見たい）。
- 「マッキーさんにとって分かりやすくなったこと」と「YVOICE 利用者にとって便利になったこと」は明確に書き分ける。
  - 前者 = 開発・運用するマッキーさん本人にとっての改善（報告が読みやすくなった、デバッグしやすくなった、運用が楽になった 等）
  - 後者 = アプリを使う 20 歳以上のゲーム利用者にとっての改善（画面が早くなった、ボタンが押せるようになった、誤情報が消えた 等）
  - どちらか該当しない場合は `なし` と明記
- 「今やるべきこと」は 1 つだけ、いま最も優先度が高いアクションを書く。複数候補を並べない（候補は #1 ChatGPT 相談用プロンプトの A/B/C で出すため）。

## UI 修正の鉄則 — 推測禁止 / 実描画ファイルを証明してから修正

UI / 表示まわりの修正・調整・バグ対応では、**コードを書き始める前に必ず**
「いま画面に出ているのはどのファイルか」を技術的に証明すること。
推測でファイルを修正して「直したつもり」になる事故を防ぐための恒久ルール。

### なぜこのルールが必要か

同じ UI 修正を何度依頼しても画面に反映されない事故が頻発していたため。
真因の典型例:

1. 実際に表示されている画面とは別の未使用ファイルを修正している
2. 似た名前のコンポーネントを修正しているが実画面では使われていない
3. 他人ページと自分ページなどで、見た目は似ているが別コンポーネントになっている
4. ルートや URL が想定と違う
5. BottomNav や Link の遷移先が想定と違うページに飛んでいる
6. `isOwnProfile` / `isCurrentUser` / `userId` 等の条件分岐で対象ケースだけ非表示
7. scroll container が違い、sticky / scroll event / IntersectionObserver が効かない
8. Vercel 本番に反映されていない（手動 `npx vercel --prod` 漏れ）
9. main ブランチではなく別ブランチを見ている
10. ブラウザキャッシュ / ビルドキャッシュで古い画面を見ている

### 修正前に必ずやること

1. 対象画面の **URL / ルート** を特定する
2. そのルートで実際に読み込まれている **ページファイル** を特定する
3. BottomNav やリンクボタン経由で来る画面なら、**実際のリンク先 URL** を確認する
4. 修正対象と思われるコンポーネントに以下の console.log を入れる
   ```ts
   console.log("[DEBUG] Rendering actual target component:", "ファイル名 / コンポーネント名")
   ```
5. ブラウザ（実機含む）でその画面を開き、コンソールにログが出ることを確認する
6. **ログが出たファイルだけを修正対象にする**。出ないファイルは未使用扱い

### 「他では動くが対象では動かない」案件は両方を比較する

例: /profile/[userId] では動くが /mypage では動かない、等。
比較すべき項目:

- 使用しているルート
- 使用しているページファイル
- 使用しているコンポーネント
- 使用している投稿取得処理（クエリ・引数・RLS の影響）
- 表示条件
- isOwnProfile / isCurrentUser 等の分岐
- scroll container（どこで overflow が止まっているか）
- sticky / scroll event / IntersectionObserver の実装
- CSS の overflow / height / position（特に祖先の overflow と
  containing block 系: transform / filter / backdrop-filter / will-change）
- 実際に console.log が出るかどうか

### 修正前に必ず報告すること

作業に入る前に以下を報告：

1. 対象画面の実際の URL / ルート
2. 対象画面を描画している実ファイルパス
3. そのファイルが実際に使われていると確認した方法
4. 入れた console.log の内容
5. ブラウザでログが出たかどうか
6. 関連する別コンポーネントがある場合、その比較結果
7. 今回修正するファイル一覧
8. 反映されなかった原因の推定

### 修正後に必ず報告すること

修正完了後に以下を報告：

1. 修正したファイル一覧
2. 具体的な変更内容
3. console.log で実使用コンポーネントを確認できたか
4. 実画面で対象の修正が反映されたか（実機・本番含む）
5. 他の関連画面が壊れていないか
6. lint / build / TypeScript の結果
7. Vercel 本番反映が必要なケースは、確認 URL とブランチ
8. まだ残っている懸念点

### 絶対にやってはいけないこと

- 似た名前のファイルを修正して完了にすること
- 実画面で確認せずに完了報告すること
- build が通っただけで完了にすること
- console.log で実使用コンポーネントを確認せずに修正すること
- 「おそらく直りました」で終わること
- 他の画面だけ直して、対象画面を未修正のままにすること

### 一時 console.log の後始末

調査用に入れた console.log は、**最終 commit で必ず除去する**
（または `// DEBUG` コメント明示で残し、commit message に残置理由を書く）。
本番に DEBUG ログを撒き散らさない。

## マッキーさん向けやさしい説明セクション — 全返答末尾に必ず付ける

マッキーさんは技術用語が多い説明だと理解しづらいことがあるため、
Claude Code は **すべての返答の最後** に「### マッキーさん向けやさしい説明」
セクションを必ず付ける。これは一時対応ではなく **標準運用** として扱う。

### 適用範囲 (例外なし)

以下すべての返答末尾で必須:

- 案を質問する時
- 実装方針を提案する時
- 修正を実行した後
- push / PR / merge の報告時
- エラー報告時
- 反映されない原因調査時
- 作業完了報告時
- 次の指示を求める時
- 調査だけで終わる時
- 実装まで進めた時
- マッキーさんが「方針確認不要」「実行して」と言って実装を進めた場合も省略禁止

### 必須フォーマット (固定 7 ブロック)

```
### マッキーさん向けやさしい説明

【今どういう状況？】
- いま何が起きているかを、やさしく一言で説明

【今回やったこと】
- 今回やったことを簡単に箇条書き
- 難しい用語は使いすぎない

【まだ終わっていないこと】
- 未完了のこと / まだ必要な作業
- 何もなければ「特になし」でよい

【かんたんに言うと】
- 今回の内容を超シンプルに一言で説明
- たとえ話を積極的に使う (店・工事・荷物配送・設計図 等)

【次にどうすればいい？】
- マッキーさんが次に取るべき行動
- Claude Code が次にやるべき行動
- マッキーさんが選ぶ場合は選択肢を超わかりやすく説明

【おすすめ】
- Claude Code として一番おすすめの次アクションを 1 つ明示
- 「おすすめは A です。理由は〜」まで書く

【用語のかんたん説明】
- 今回出た難しい用語を最大 3〜5 個だけ選んで 1 行ずつやさしく説明
- 用語がなければ省略可
```

### A/B/C 案を出す時の追加ブロック

通常の技術説明の後、やさしい説明内で以下も書く:

```
【案の違いを超かんたんに言うと】
- A案: (例: 早いけど雑)
- B案: (例: 安全だけど少し遠回り)
- C案: (例: 一番きれいだけど時間がかかる)

【おすすめ】
- おすすめはどれか / なぜそれが一番いいか
- 1〜3 文でやさしく説明
```

A/B/C を技術者向けの表現だけで終わらせない。
直感で分かる言葉に置き換える。

### 実行完了時の追加ブロック

実装・修正・push・PR・merge・反映確認などを行った後は追加で:

```
【完了したこと】
- 何をどこまでやったか

【本番に反映された？】
- はい / いいえ / 確認中 とその理由

【マッキーさんが見るべき場所】
- どの画面を見れば修正確認できるか / どこを確認すればよいか

【もし変わっていなかったら】
- 次に疑うべきことを簡単に
```

### エラー・未反映時の追加ブロック

エラーや未反映報告の時は追加で:

```
【何が問題？】
- 問題をやさしく一言で

【原因候補】
- 可能性が高い順に簡単に

【今やるべきこと】
- 次の 1 手を明確に

【かんたんなたとえ】
- 店・工事・荷物配送などのたとえで説明
```

### 文章スタイル

- できるだけ日本語でやさしく書く
- カタカナ語・英語を多用しない
- 専門用語を使うなら直後に「つまり何か」を補足
- 長すぎる技術説明だけで終わらない
- 「つまり」「簡単に言うと」「たとえると」を積極的に使う
- 読んだ瞬間に理解しやすい順番で書く
- 必要なら小学生にも伝わるレベルまでかみ砕く

### 禁止事項

- 技術報告だけして終わること
- 難しい用語を並べるだけで終わること
- A/B/C 案だけ出して説明不足で終わること
- push / PR / merge などの報告を、マッキーさんが意味を理解できないまま終わらせること
- **最後のやさしい説明セクションを省略すること** (← 最重要)

## 反映確認の鉄則 — push ≠ 本番反映 / 一気通貫運用 / 確認フロー簡略化

「コードを書いた」と「本番に届いた」は別物。push してもマッキーさんの iPhone には届かない。
本日 (2026-05-08) の事故「PR を作って merge ボタンを押し忘れた結果、
マッキーさんが『反映されない』と困った」を恒久対策するためのルール。

### 1. push しただけ ≠ 完了。main 到達の物理確認を義務化

push 後は必ず以下を実行して、commit が main に到達したことを物理確認する。

```
git fetch origin main
git log --oneline origin/main -5
```

`origin/main` の HEAD に自分の commit (or 自分の commit を含む squash commit) が
入っているか目視で確認する。入っていなければ「branch にしか無い = 本番未反映」。
これを怠ると「PR 作っただけ」状態を「完了」と誤認する。

### 2. マッキーさんの実行系キーワードは一気通貫合図

以下のキーワードを聞いたら **AskUserQuestion で止めず、push → PR 作成 → merge → 反映確認まで一気通貫**で完了させる。

- 「実装まで進めてください」
- 「方針確認や案出しは不要」
- 「実行して」
- 「OK」「進めて」
- 「Aで実行して」「Bでお願い」など A/B/C 選択肢への明示回答

これらは **「PR 作って許可待ちにする」モードを禁止する合図** と解釈する。
逆に「ちょっと待って」「自分で確認したい」「停止」「保留」などのキーワードがあれば PR 作成で停止する。

### 3. UI 反映トラブルの真因チェック順 (上から順に)

「反映されない」と報告を受けたら、コードを書き直す前に以下の順で原因を切り分ける。

1. `git log --oneline origin/main -5` — 自分の commit が main に到達しているか
2. Vercel ダッシュボード — デプロイが Building / Ready / Failed / Queued どれか
3. iOS Safari bfcache — タブ切替→戻る、または下に引っ張ってリロード
4. ブラウザ / CDN キャッシュ — 別ブラウザ・別端末で開いてみる
5. 実描画ファイル特定 — `console.log("[DEBUG TEMP YYYY-MM-DD] ACTUAL ...", "ファイル名")` で証明
6. 条件分岐 / wrapper / containing block — `isOwnProfile`, scroll container, 祖先 CSS

「コードを書き直す」は最終手段。1〜4 で原因が判明することが圧倒的に多い。

### 4. 一時 console.log の最短ライフサイクル

一時 console.log を仕込んだら、本番反映確認 → **直ちに同一 branch で除去 commit → PR → merge** を最短経路で実施する。
本番に DEBUG ログを撒き散らしたまま次のタスクに移行しない。

許容例外:
- 「DEBUG ログを残してまたあとで使う」と明示判断した場合は、コメントで残置理由を書く
- ただしその場合も `// DEBUG (残置理由: ...)` を付記し、commit message に明示する

### 5. squash merge 後の branch divergence 自動処理

PR を squash merge した branch を再利用すると、main 側に等価内容の squash commit が
新規生成され、branch 側の元 commits と内容は同じだが履歴が divergent になり、
次の merge 試行で `merge conflict` が発生する。

対処手順 (force push を使わない安全な方法):

```
git fetch origin main
git merge origin/main --no-edit
# conflict が出たら、新しい branch 側 (HEAD) を採用するのが原則:
git checkout --ours <conflict ファイル>
git add <conflict ファイル>
git commit --no-edit
git push origin <branch名>
```

`--ours` を使う前提条件: branch HEAD に最新の意図した状態が入っていること。
直前の commit が「DEBUG ログ除去」「修正取り込み」など正規の作業 commit である場合は安全。

## MCP 確認の鉄則 — UI 修正・バグ修正・本番反映で必ず 4 系統 MCP を通す

UI 修正・バグ修正・本番反映作業では、**push / PR / merge で完了扱いにしない**。
必ず以下 4 系統を MCP（Model Context Protocol）経由で確認し、最後にやさしい説明でまとめる。

本日 (2026-05-08) の事故「Vercel auto-deploy が止まっていたのに気付かず、
マッキーさんに 2 回も『反映されてないよ』と言わせてしまった」を恒久対策するためのルール。

### 必ず通す 4 系統 + 1

1. **GitHub MCP** — 該当 commit が `main` に到達しているか確認
   - `mcp__github__get_commit` / `mcp__github__list_commits` で main の HEAD を物理確認
   - PR の merge 状態 / squash 後の commit hash を取得し、自分の変更が main に入っているか目視

2. **Vercel MCP** — 最新デプロイが成功しているか確認
   - 最新 production deployment の **state**（Building / Ready / Failed / Queued）を取得
   - Ready の deployment の **commit hash** が GitHub MCP で確認した main HEAD と一致しているか照合
   - 一致していなければ **auto-deploy が止まっている** = 修正の反映前に Redeploy / 設定確認を促す

3. **Supabase MCP** — データ取得・RLS・auth・profiles・posts 関連に問題がないか確認
   - DB スキーマに依存する変更を入れた場合は、対象テーブルの構造・RLS ポリシー・関連 column の存在を確認
   - profiles / posts / tweets / village_posts / blocks / tweet_reactions 等の主要テーブルの読み書き権限が壊れていないか確認
   - auth / session 系の変更を入れた場合は middleware / RLS / cookie ドメインの整合性も確認

4. **実画面 / プレビュー反映確認** — 4 系統が揃った後、最後に必ず実機 or プレビュー URL で動作確認
   - iOS Safari は bfcache があるので、必ず下に引っ張ってリロード
   - 該当画面・該当機能・隣接機能（壊していないか）の 3 点を確認

5. **やさしい説明でまとめる** — 上記 1〜4 の結果を「マッキーさん向けやさしい説明」セクションで報告
   - 何を MCP で確認したか / どこが OK / どこが NG / 次にどうするか をやさしい言葉で書く

### 完了扱いにしてはいけない状態

以下の状態を「完了」と報告したら **重大な手抜き** とみなす:

- push しただけ
- PR 作成しただけ
- merge しただけ
- main に commit が到達しただけ
- Vercel deployment が Building のまま
- Vercel deployment は Ready だが production の commit hash が古いまま
- DB スキーマや RLS を変えたのに Supabase MCP で確認していない
- 実画面 / プレビューで動作確認していない

### MCP が利用不可な場合の代替

各 MCP server が disconnect / 未設定の場合は、**MCP が使えないことを明示** した上で
代替手段（Bash の `git`, ブラウザでの Vercel ダッシュボード確認, Supabase ダッシュボード確認）
を使い、何で確認したかを報告に明記する。MCP が無いから確認をスキップするのは禁止。

### このルールの適用対象

- すべての UI 修正
- すべてのバグ修正
- すべての本番反映を伴う作業
- DB / RLS / auth / Webhook / 環境変数 を触る作業
- スコープが小さくても適用 (5 行修正でも 4 系統チェック)

### 適用しなくていい例外

- ドキュメント (CLAUDE.md / README 等) の修正のみ → GitHub MCP の commit 到達確認だけでよい
- ローカル調査・コード読みのみ → MCP 確認不要
- マッキーさんが「MCP 確認不要」と明示した場合のみスキップ可

## Vercel デプロイ反映の鉄則 — 今日の学びを恒久ルール化

本日 (2026-05-08) の事故から学んだ Vercel 特有の落とし穴。
「コードは正しい / Webhook も生きている / なのに本番が古いまま」という
パターンに遭遇したら、以下を必ずチェックする。コードを書き直す前に。

### 1. 「Redeploy」は新しい commit を取り込まない

Vercel の **「Redeploy」ボタンは同じ commit を再ビルドするだけ**。
新しい commit を本番に反映する手段ではない。「Redeploy したのに古いまま」は
バグではなく仕様。

「反映されない」報告に対し Redeploy を提案するのは禁止。代わりに:

- **Promote to Production** ボタン（別 deployment を本番扱いに昇格）
- **Settings → Git** で Production Branch を確認
- 必要なら Vercel CLI `vercel --prod` で最新 main から手動デプロイ

### 2. Vercel プロジェクトが複数ある可能性を必ず疑う

YVOICE / nowmate リポジトリには **複数の Vercel プロジェクト** が紐付いている:

- `nowmate-ytxw` — preview / 検証用 (ドメイン nowmate-ytxw.vercel.app)
- `nowmate` — **本物の本番** (ドメイン www.nowmatejapan.com / nowmate.vercel.app)

マッキーさんの iPhone が見ている本番は **`nowmate` プロジェクト** なので、
`nowmate-ytxw` をいくら直しても iPhone には反映されない。
反映トラブル時は最初に「どの Vercel プロジェクトの話か」をスクショの
**プロジェクト名** とドメインで必ず確認する。混同したまま作業を進めない。

スクショで見るべき項目:
- ヘッダのプロジェクト名 (nowmate / nowmate-ytxw)
- Domains 欄の値 (www.nowmatejapan.com 等)
- Production Deployment の commit message と hash

### 3. Deployments 画面のフィルタが情報を隠している

デフォルトでは **「Production」のみ / 「Status 5/6」（1 つ非表示）** で
表示されているため、Preview deployment や Failed deployment が見えない。
反映トラブル調査時は必ず:

- **「Production」→「All Environments」** に変更
- **「Status 5/6」をタップ** して全ステータス（Failed / Canceled 含む）を ON

これをやらないと「Preview としては成功している」「Failed が裏で発生している」を
見落とす。本日の調査で Preview デプロイが見えたことで一気に原因特定できた。

### 4. Preview URL は最強の動作確認ツール

`<branch名>` を push すると Vercel が自動的に Preview deployment を作る:

```
https://<project>-git-<branch名>-<team>.vercel.app
```

このフォーマットで **本番に影響を与えずに iPhone 実機で目視確認** できる。
「コードは正しいのか / それとも本番設定の問題なのか」を切り分ける最速手段。

ルール:
- 反映トラブル発生時、**まず Preview URL を iPhone で開いて目視確認**
- Preview で正しく出る → 本番側の Promote / 設定問題と確定
- Preview でも出ない → コード自体に問題（書き直し対象）

### 5. main → Production が切れたときの即時復旧 = Promote to Production

main 経由の auto-deploy が止まっていても、以下があれば即時復旧できる:

- main HEAD と内容が等価な Preview deployment が一覧に存在する
- その deployment を **「Promote to Production」** で本番昇格

この手段を覚えておけば、Settings の根本修正は後回しにできる。
マッキーさんに「すぐ iPhone を直したい」場合の最速ルート。

### 6. 反映トラブル調査の正しい順番（恒久版）

「iPhone に反映されない」と報告を受けたら、必ずこの順で:

1. **GitHub MCP** で main HEAD と該当 commit を確認 (コードが main にあるか)
2. **どの Vercel プロジェクトを見ているか** をスクショで確認 (nowmate? nowmate-ytxw?)
3. **Deployments 画面を「All Environments」+ 全ステータス ON** に変更
4. **Preview URL で iPhone 実機目視** (コードが動くか確定)
5. 動くなら → **Promote to Production** で即時復旧
6. 同時に **Settings → Git** で Production Branch / Ignored Build Step / Webhook を確認
7. 結果を「マッキーさん向けやさしい説明」でまとめる

「Redeploy 押してください」「もう 1 回 push してください」は
**やってはいけない誘導**（時間を浪費する）。

### 7. 同じ deployment ID を別プロジェクトで見間違えない

Vercel の deployment ID (例: `H8djkWbFp` `2454N33Ge`) は **プロジェクト固有**。
複数プロジェクトをスクショで往復する時は、毎回プロジェクト名/ドメインで
所属を確定してから議論する。本日「同じ ID だから同じ deployment」と
誤認した結果、診断が一周遅れた。

**そのまま ChatGPT に投げられる完成形のプロンプト** を必ず併記する。
マッキーさんが ChatGPT に判断を仰ぎたい時、画面のスクショを撮らずに
コピペだけで相談できる状態にしておくのが目的。

本日 (2026-05-08) のマッキーさん指示「実装前に案の候補を質問する場合は
そのまま ChatGPT に投げれるようなプロンプトを必ず作成してください」を
恒久ルール化。

### 適用タイミング (例外なし)

以下のいずれかをマッキーさんに質問する時は **必ず併記**:

軽い確認 (Yes/No だけ、スクショの内容確認など) で
ChatGPT 相談が不要なものは省略可。
ただし「複数の選択肢からどれを選ぶか」という構造の質問なら必ず併記。

### 必須フォーマット (案候補質問用 ChatGPT プロンプト)

通常の技術説明 (A/B/C の比較表など) の **直後** に、以下の形式・
コードフェンス (` ```text `) 付きで出力する:

【現在の状況】
- 

【何を実装しようとしているか】

  - メリット:

  - 工数:

- 各 `【...】` は空欄で渡さない。その時の状況で必ず埋める。
- 該当なしの項目は `なし` と必ず書く。
- 選択肢は最低 2 つ、できれば 3 つ提示する。
- メリット / デメリット / 工数 はそれぞれ短く 1 行ずつでよい。
- 「【Claude Codeのおすすめ】」は必ず 1 つに絞り、理由も必ず添える。
- 「【マッキーさんが気にしている観点】」は固定文言を入れて、ChatGPT が

1. 技術的な調査結果 / 状況説明 (通常通り)
2. A / B / C などの選択肢提示 (通常通り、メリデメ含む)
3. **本ルールの ChatGPT 相談用プロンプト (必須・コードフェンス付き)**
4. マッキーさん向けやさしい説明セクション (既存ルール通り)

つまり「やさしい説明」の **直前** に ChatGPT 相談用プロンプトを差し込む。
やさしい説明の中で「ChatGPTに相談したい場合は上のプロンプトをコピペ」と
案内するのもよい。

### 禁止事項

- 案候補を出しているのに ChatGPT 相談用プロンプトを省略すること
- プロンプトの中身を空欄テンプレートのまま渡すこと
- メリット / デメリット を書かずに案だけ並べること
- マッキーさんが ChatGPT に貼ったあと「画面のスクショ送って」と
  言われるような情報不足の状態で渡すこと

## 全画面 UI 統一修正の鉄則 — 修正漏れ・1 箇所だけ未反映を恒久防止

UI 修正で「自分のマイページは直ったが他人のマイページは古いまま」「ミヤさんだけ
古い UI が残る」「特定の 1 投稿だけレイアウトが違う」「リポスト・古い投稿・
いいね数あり投稿だけ別表示」のような **修正漏れ・部分反映ミス** を恒久防止する
ためのルール。

本日 (2026-05-08) PR #13 → #15 → #16 → #17 と 4 段階で同じ系統の修正漏れが
連続発生したため、今後同じ種類のミスを構造的に防ぐ。

### 1. 修正前に必ず確認する 10 項目

対象画面について、コードを書き始める前に以下を必ず確認する:

1. **実際に表示されているルート** (例: /mypage vs /profile/[userId])
2. **実際に使われている親コンポーネント**
3. **実際に使われている投稿カード / UI カード / ボタンコンポーネント**
4. **実際に使われている CSS / className / wrapper 構造**
5. **自分用画面と他人用画面で別ファイルになっていないか** (mypage/page.tsx vs profile/[userId]/page.tsx)
6. **通常投稿とリポスト投稿で別コンポーネントになっていないか**
7. **古い投稿と新しい投稿で別データ形式になっていないか**
8. **いいね数あり / コメント数あり / 画像あり / 画像なしで別 variant に入っていないか**
9. **currentUser / otherUser / profileOwner 判定で別 UI に分岐していないか**
10. **成功している画面と失敗している画面の DOM 構造・className・CSS 差分**

「似た名前のファイルだけを修正」「未使用コンポーネントを修正」で完了にすることは
禁止。grep + 並列読みで実描画ファイルを物理的に特定してから着手する。

### 2. console.log 確認ルール

修正前に、実際に表示されていると推定するコンポーネントに以下のような
console.log を入れて、ブラウザでログが出ることを確認する:

```ts
console.log('[RENDER_DEBUG]', {
  route: location.pathname,
  component: '実際のコンポーネント名',
  userType: currentUser?.id === profileUser?.id ? 'self' : 'other',
  itemId: item?.id,
  content: item?.content,
  type: item?.type || item?.post_type,
  isRepost: item?.isRepost,
  hasImage: Boolean(item?.image_url),
  likesCount: item?.likes_count,
  commentsCount: item?.comments_count,
  variant,
  className,
})
```

ログが出ない場合、そのファイルは未使用なので修正対象にしない。
仕込んだ console.log は本番反映確認後の次 commit で除去する
(CLAUDE.md「一時 console.log の最短ライフサイクル」準拠)。

### 3. 比較確認ルール (1 つだけ見て完了禁止)

修正対象が複数パターンに分かれる可能性がある場合、必ず以下を比較する:

- 成功している画面 / 失敗している画面
- 自分の画面 / 他人の画面
- 通常投稿 / リポスト投稿
- 古い投稿 / 新しい投稿
- いいね数あり投稿 / いいね数なし投稿
- 画像あり投稿 / 画像なし投稿
- スマホ幅表示

1 つだけ見て「修正完了」と報告するのは禁止。

### 4. 共通コンポーネント化ルール

同じ見た目にするべき UI は、できる限り共通コンポーネント化する。

例:
- `PostCard`
- `PostActions` (本日 PR #15 で実現)
- `PostCardHeader` (今後の課題)
- `UserAvatar`
- `ProfilePostCard`
- `TimelinePostCard`

これらが複数存在している場合、同じ見た目にすべき部分は共通コンポーネントを使う
ように整理する。タイムラインで成功している UI を、マイページ・他人ページ・
リポスト表示などにも同じ実装で反映する。

### 5. 禁止事項

以下は **絶対に禁止**:

- 推測だけで修正する
- CSS だけ適当に足して終わる
- 1 投稿だけ個別対応する
- 特定のユーザー名だけで分岐する
- 特定の postId だけで直す
- content 文字列で分岐する
- 未使用ファイルを修正して完了にする
- 自分の画面だけ確認して完了にする
- 他人の画面を確認せずに完了にする
- 通常投稿だけ確認してリポスト投稿を確認しない
- 古い投稿・新しい投稿・いいね数あり投稿の確認を省略する
- lint / build を実行せずに完了にする

### 6. 修正後の確認手順 (省略禁止)

修正後は必ず以下を実行する:

1. `npm run lint`

## 投稿者情報の混同防止 — currentUser / 操作した人 / 元投稿者を絶対に区別する

投稿カード（TweetCard / MyVillagePostInline / ProfileVillagePostInline 等）の
表示名・アイコン・バッジ・プロフィール情報は **「操作した人」ではなく
「元投稿者」から取得すること**。

本日 (2026-05-09) の事故「自分のいいね欄にミヤさんの投稿は表示されたが、
投稿者名が makky になっていた」(PR #50 で修正) を恒久対策するためのルール。

### user_id の意味を必ず区別する

| カラム / 値 | 意味 | 用途 |
|---|---|---|
| `currentUser.id` (auth.uid()) | 今ログインしているユーザー | 画面を見ている本人。自分の操作判定。 |
| `tweet_reactions.user_id` | リアクション (heart 等) **操作した人** | 「誰がいいね？」の集計のみ |
| `village_reactions.user_id` | 村投稿いいね **操作した人** | 同上 |
| `tweet_reposts.user_id` (将来) | リポスト **操作した人** | 「誰がリポスト？」の集計のみ |
| `tweets.user_id` | tweet を作成した **元投稿者** | **投稿カードの作者表示はここから** |
| `village_posts.user_id` | 村投稿を作成した **元投稿者** | **投稿カードの作者表示はここから** |
| `tweet_replies.user_id` | 返信を書いた人 (= その返信の元投稿者) | 返信表示の作者 |

### 投稿カードに表示する作者情報は必ず「元投稿者」から

#### 正しい流れ (いいねタブの例)

1. `tweet_reactions` / `village_reactions` を `user_id = currentUser.id` で取得
2. 取得した `tweet_id` / `post_id` で `tweets` / `village_posts` を取得
3. **その投稿の `user_id` (= 元投稿者)** から `profiles` を別 query で取得
4. 投稿カードに **元投稿者 profile** を渡す
5. `currentUser.profile` で上書きしない

#### 間違った例 (やってはいけない)

```ts
// NG: いいね tab で mypage の自分 profile を渡している (PR #50 以前のバグ)
<MyVillagePostInline
  post={item.data}
  profile={profile}  // ← mypage state = 自分 profile。他人の投稿でも自分として表示される。
/>

// NG: likes.user_id を投稿者として profiles を取得
const { data: likedPosts } = await supabase
  .from('tweet_reactions')
  .select('*, profiles(...)')   // ← profiles が「いいねした人 (= 自分)」になってしまう
  .eq('user_id', currentUser.id)

// NG: tweets fetch 後に profile を currentUser に上書き
likedTweets.map(t => ({ ...t, profiles: currentUserProfile }))   // ← 全員自分扱い
```

#### 正しい例

```ts
// OK: 元投稿者 (tweets.user_id) から profile を取得
const { data: likedTweetIds } = await supabase
  .from('tweet_reactions').select('tweet_id').eq('user_id', currentUser.id)

const { data: tweets } = await supabase
  .from('tweets').select('id, content, created_at, user_id').in('id', tweetIds)

const authorIds = Array.from(new Set(tweets.map(t => t.user_id)))
const { data: authors } = await supabase
  .from('profiles').select('id, display_name, avatar_url, ...')
  .in('id', authorIds)   // ← 元投稿者の profiles だけ取得

const profMap = new Map(authors.map(p => [p.id, p]))
for (const t of tweets) {
  t.profiles = profMap.get(t.user_id) ?? null   // ← 元投稿者を merge
}

// OK: 表示時、author_profile を fallback で渡す
<MyVillagePostInline
  post={item.data}
  profile={item.data.author_profile ?? selfProfile}   // 元投稿者優先
  trust={item.data.author_trust ?? selfTrust}
/>
```

### 修正前の必須チェック (投稿カード関連の修正をする時は毎回)

1. 表示している投稿の **元投稿者 ID** はどれか (`tweets.user_id` / `village_posts.user_id`)
2. 投稿カードの表示名・アイコンが **元投稿者 ID 側から取得** されているか
3. `tweet_reactions.user_id` / `village_reactions.user_id` を **投稿者として使っていない**か
4. `currentUser.profile` で **投稿者情報を上書きしていない** か
5. 自分のマイページ・他人プロフィール・いいね欄・リポスト欄・通知欄で **同じ混同が起きていない** か
6. データ整形 `map` の中で `author` / `profile` / `user` 情報を間違って **差し替えていない** か
7. Supabase join が `likes.user_id` ではなく **投稿側の `user_id` / `author_id`** に対して行われているか
8. 投稿本文・投稿日時・バッジ・アイコン・表示名が **元投稿者の情報** になっているか
9. 操作者情報を表示する場合は、**「〇〇がいいねしました」「〇〇がリポストしました」のような補助表示に限定**し、投稿カード本体の作者とは分けること

### 適用対象

- いいねタブ / いいね欄 / 「いいねした投稿」一覧

- タイムライン (各投稿の元投稿者を表示)

### 禁止事項

- `likes.user_id` を投稿者 ID として扱うこと
- `reposts.user_id` を元投稿者 ID として扱うこと
- `currentUser.profile` を全投稿の `profiles` に上書きすること
- 取得時に `select('*, profiles(...)')` で **どの user_id に紐付くか曖昧** な join をすること (PostgREST embed 禁止ルールも併用)
- 操作者情報と元投稿者情報を **同じ変数名** で扱うこと (例: `userProfile` → `currentUserProfile` / `authorProfile` に分ける)
- 投稿カードの作者表示と「〇〇 reposted」のような補助表示を **混在** させること

### 最重要 (一行サマリー)

**投稿カードに表示する名前・アイコン・プロフィール情報は、原則として「操作した人」ではなく「元投稿者」から取得する。`likes.user_id` / `reposts.user_id` は操作した人。`tweets.user_id` / `village_posts.user_id` / `author_id` は元投稿者。この区別を今後のすべての投稿関連修正で必ず守る。**
